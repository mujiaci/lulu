# 辞海长期记忆沉淀设计

## 目标

辞海继续承担角色可见的日记、心迹、阅读和反思记录；长期记忆只保存以后确实值得想起、且有原始证据支撑的压缩摘要。两者不再一条记录对应一条向量记忆。

本阶段解决以下问题：

- 停止 `CihaiService` 对每条辞海内容立即写入 `MemoryBank`。
- 为每个角色建立互相隔离、可恢复的待沉淀队列。
- 批量提炼、去重并校验记忆候选，拦截系统模板、工具 JSON 和空泛自我反思。
- 保持辞海展示、删除和阅读流程不依赖长期记忆服务是否可用。
- 在进程中断或模型失败后安全重试，不重复生成已经落库的记忆。

## 非目标

- 不重写聊天消息的长期记忆提取流程。
- 不改变向量模型、召回排序或 MemoryBank 数据库结构。
- 不把所有辞海记录都强行转成长期记忆。
- 不在本阶段增加新的设置页面或要求用户手动管理队列。

## 数据边界

`CihaiState.entries` 是用户可见记录的事实来源。新增的 `memoryQueue` 只保存待处理条目的 ID、角色 ID、入队时间、重试次数和下次重试时间，不复制正文。

每个 `CihaiEntry` 保留旧 `memorySaved` 字段用于兼容已有数据，并增加明确的沉淀结果：

- `PENDING`：仍在队列等待批量处理。
- `SAVED`：该条目成为至少一条长期记忆的证据。
- `CIHAI_ONLY`：已审阅，但没有足够质量写入长期记忆。

旧数据中 `memorySaved=true` 视为 `SAVED`；其余旧条目不自动全量入队，避免升级后把历史噪声集中写入记忆库。只有新产生的辞海条目进入队列。

## 沉淀流程

1. 新辞海条目通过一次原子更新写入可见列表并加入对应角色队列。
2. 同一角色的待处理条目达到批量阈值，或最老条目超过等待上限时，选择一个有上限的批次。
3. 调用角色的记忆提取模型；优先使用 `memoryEmbeddingConfig.extractionModelId`，否则使用角色聊天模型或全局聊天模型。
4. 提示词只允许提取以下长期价值：
   - 新的稳定用户事实、偏好和边界。
   - 明确承诺及执行结果。
   - 有用户行为或话语证据的关系变化。
   - 高情绪权重的共同事件。
   - 用户主动纠正的信息。
5. 本地质量门验证候选类型、正文、证据 ID、角色归属和噪声标记，并对同一批次的近似文本做规范化去重。
6. 合格候选批量写入 `MemoryBank`。候选的 `sourceMessageNodeIds` 使用 `cihai:<entryId>`，作为幂等证据键。
7. 成功后移除本批队列项：被候选引用的条目标记 `SAVED`，其余标记 `CIHAI_ONLY`。
8. 失败时不改变辞海内容，队列项按有限指数退避更新；下次新增记录或应用启动维护时继续处理。

## 质量规则

长期记忆必须同时满足：

- 内容是当前角色第一人称的压缩记忆，而不是旁白或架构说明。
- 至少引用一条当前批次的辞海证据 ID。
- 对用户偏好、承诺、纠正、边界和关系判断给出具体用户信号。
- 关系变化不能只由系统自己的情绪或推断生成。
- 内容具有未来召回价值，不是对本轮流程的复述。

以下内容直接拒绝：

- 通用 fallback 心声和没有新事实的重复担心。
- 工具 JSON、工具状态、请求参数和 observation 原文。
- 感知层、评估层、判断层、七层架构等系统字段说明。
- “我整理了记忆”“之后可以参考”一类无具体事实的元反思。
- 仅由阅读材料产生、没有连接到用户事实或共同经历的泛化结论。
- 无用户证据的亲密度、信任或关系升级。

## 隔离与幂等

- 每次批次选择只读取一个 `assistantId`。
- 模型返回的证据只能引用该角色当前批次中的条目。
- 长期记忆继续写入对应 `assistantId`，不会进入其他角色空间。
- 写入前从 MemoryBank 读取已有 `cihai:<entryId>` 证据，跳过已沉淀条目；因此即使进程在“写库成功、清队列前”中断，也不会再次写入相同批次。

## 体验表现

辞海页面不再显示含糊的“待记忆”。状态文案改为：

- `待沉淀`
- `已沉淀`
- `仅保留在辞海`

这让用户能区分“角色写下了这件事”和“系统认为它值得进入长期记忆”。删除辞海条目时同步移除尚未处理的队列项，但不追溯删除已经形成的长期记忆。
