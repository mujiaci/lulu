# Lulu 核心瘦身与活人感审计

> 审计目标：减少重复编排、巨型文件和不可解释的后台分支，同时保留并强化陪伴、娱乐、学习监督和生活监督。  
> 审计依据：`AGENTS.md`、`COMPANION_REBUILD_PLAN.md`、代码索引 schema v3、核心调用链实现与当前 CI 记录。  
> 原则：先建立可回归的统一主链路，再拆分和删除；不得用“精简”掩盖功能退化。

## 结论

Lulu 的“活人感地基”已经明显强于普通聊天壳：

- 有连续记忆、滚动摘要、关系证据、承诺、关注、生活事件和真实工具结果；
- 聊天、电话、主动消息能够共享角色人设、世界书和部分运行时状态；
- 主动联系不是固定次数轮询，能够结合离线、承诺、用户忙碌和事件状态；
- 数字生活活动要求真实记录，原则上不允许角色声称完成不存在的行动；
- 上下文有分场景预算、去重和不可静默裁剪的固定层。

当前主要问题不是“功能太少”，而是**同一套陪伴逻辑分散在几个超大协调器里**。这会产生三个直接后果：

1. 聊天、电话、主动消息和游戏在后处理细节上逐渐分叉；
2. 每次增加一种监督或娱乐能力，都要在多个大文件重复接线；
3. 私有状态里存在少量固定兜底措辞，容易让冷淡、严厉、敌对或非人角色最终都呈现相似的默认陪伴人格。

因此，项目应该做“架构减脂”，而不是大规模删除产品能力。

## 代码索引现状

索引已经升级为三层：

1. `CODE_INDEX.md`：给人读的模块、核心链路和热点导航；
2. `code-index.json`：逐文件符号、SHA-256、类别、测试属性和 TODO 信号；
3. 反向依赖数据：明确 import、已解析目标、`importedBy` 和完整符号索引。

它可以显著减少以下工作：

- 定位一条产品链路的主要入口；
- 判断一个文件是核心依赖还是边缘能力；
- 删除或移动文件前查找明确的反向依赖；
- 优先识别超大文件、高导入扇出和高扇入公共模型；
- 区分生产代码、测试和季节性/临时逻辑。

它仍然不是编译器调用图，无法完整识别同包调用、反射、Koin 注册、序列化名称、插件事件和动态工具注册。任何删除仍须回读实现、DI、数据库、测试与失败路径。

## 产品核心边界

以下能力视为产品内核，精简时不得删除：

- 角色核心人设、关系类型、世界书和用户明确边界；
- 连续记忆、滚动摘要、用户纠正覆盖旧推断；
- 聊天、电话、主动消息、游戏之间的事实与立场连续性；
- 承诺、关注、履约结果和失败/重试历史；
- 学习、起床、睡眠、健康和时间事项的真实监督闭环；
- 有真实执行证据的收藏、日记、游戏、回放和共同计划；
- 工具执行结果与生活事件，禁止虚构已经完成的动作；
- 回复先展示，非关键记忆提取和维护任务后台执行。

可延后、按需加载或作为插件处理的能力：

- 与核心陪伴无关的文档、搜索、Web 管理和开发者页面；
- 低频娱乐入口和实验性插件；
- 只提供一次性 UI 展示、没有连续性价值的装饰功能。

“可延后”不等于立即删除。必须先用反向依赖、入口可达性、真实使用记录和 APK 体积数据证明其成本。

## P0：先恢复可信的工程基线

### 目标

- `assembleRelease` 编译、打包、签名成功；
- 核心陪伴单元测试通过；
- 旧 CI issue 与当前源码分离，不把历史失败当成当前错误；
- 建立可逐步收敛的 Lint 基线，而不是长期完全忽略。

### 动作

1. 每次只处理最新提交对应的失败日志；旧 issue 构建成功后关闭或标记过期。
2. Release 工作流保留 Kotlin 编译、测试、打包和签名硬失败。
3. 将现有 Lint 错误生成基线文件；新错误不得增加，旧错误分批清零。
4. 为索引脚本增加最小单元测试：确定性、排除生成文件、符号识别和反向依赖。

### 验收

- 连续三次 `master` 构建成功；
- 同一源码重复生成索引不会产生新提交；
- Lint 错误总数只能下降，不能上升。

## P1：建立唯一的陪伴回合引擎

### 当前问题

`ChatService` 同时负责会话、普通回复、电话回复、工具、Planner、状态、承诺、提醒、通知、插件事件、标题、建议、压缩和记忆提取。`ProactiveMessageService` 又重复一部分上下文、工具、生成和状态落库逻辑。

### 目标结构

建立 `CompanionTurnEngine`，所有角色出口共享以下阶段：

1. `prepare`：角色、人设、世界书、记忆、运行时状态、用户资料、场景预算；
2. `plan`：是否需要工具、承诺、后续关注和表达意图；
3. `generate`：流式正文及同次产生的结构化状态；
4. `executeTools`：可执行动作与真实结果；
5. `commit`：消息、连续性、状态、承诺、关注、生活事件；
6. `scheduleBackground`：记忆提取、摘要、标题和低优先级维护。

场景仅提供策略：

- `ChatTurnPolicy`
- `VoiceTurnPolicy`
- `ProactiveTurnPolicy`
- `GameTurnPolicy`

### 验收

- 四种场景不再分别拼接同类系统提示；
- 相同角色在不同出口读取同一份 persona/context envelope；
- 工具结果、承诺和连续性只落库一次；
- 任何失败不会出现“消息已显示但状态未提交且无法重试”的半完成状态。

## P2：拆分两个巨型服务

### `ChatService` 建议拆分

- `ConversationSessionCoordinator`：会话 session、取消、流更新；
- `CompanionTurnEngine`：唯一回合主链路；
- `VoiceTurnCoordinator`：电话输入、分段和可见消息同步；
- `PostTurnEffects`：通知、插件事件、标题和建议；
- `MemoryMaintenanceScheduler`：后台记忆提取、摘要与历史重建；
- `ConversationMaintenanceService`：压缩、导入导出和管理操作。

### `ProactiveMessageService` 建议拆分

- `ProactiveScheduler`：AlarmManager / WorkManager / PendingIntent；
- `ProactiveDecisionEngine`：何时联系、是否打扰、优先级；
- `CommitmentExecutionService`：起床、睡眠、学习和普通提醒；
- `ProactiveDelivery`：生成、通知、打开和回复状态；
- `ProactiveStateRepository`：队列与时间戳持久化。

### 原则

先提取纯函数与接口，再迁移调用；每次提交保持行为等价。禁止一次提交重写整个服务。

## P3：消除默认人格渗漏

### 风险

当前部分私有状态和关注摘要使用固定人类化句子，例如“留意你的休息”“记着你的学习安排”。它们作为内部元数据是可接受的，但若直接进入提示或 UI，会让不同角色共享同一种温柔照料口吻。

### 改法

- 持久层保存中立结构：`kind`、`subject`、`evidence`、`goal`、`dueAt`、`status`；
- 角色展示文本在统一回合引擎中依据具体 persona 动态实现；
- 技术失败只使用系统中立提示，不伪装成角色；
- 私有 `fallbackInnerThought` 改为结构化“未完成关注/下一步”，避免固定第一人称台词；
- 用冷淡、严厉、普通朋友、亲密、敌对和非人角色做同场景快照测试。

### 验收

同一监督事件在不同角色下：

- 事实、时间和责任一致；
- 语气、称呼、距离、冲突方式明显不同；
- 不会全部变成默认温柔、恋爱或客服式关心。

## P4：让数字生活从“注册表”变成真实循环

每一种活动必须证明完整链路：

`触发证据 → Selector → Executor → 真实产物/执行记录 → LifeEvent → 后续相关场景自然引用`

需要逐项验证：

- 私人日记是否真的产生内容，而非仅保存调用方提供的 summary；
- 未发送便签是否有真实 artifact；
- 小游戏和回放是否引用真实对局 ID；
- 收藏整理是否引用真实收藏；
- 共同计划是否能取消、完成和过期；
- `usesModel` 是否对应实际执行器，而不只是注册表元数据。

没有执行器、没有入口且没有持久化价值的活动应删除；有价值但尚未接线的活动先标为不可用，不允许角色声称完成。

## P5：学习监督去除临时覆盖层

`CurrentWeekStudyRecovery.kt` 是紧急计划覆盖层，包含具体日期，并通过可变类型转换修改 `ExamStudyPlan` 的集合。它不适合作为长期架构。

建议建立：

- `StudyPlanRepository`：唯一计划来源；
- `StudyPlanOverlay`：有明确起止日期、优先级和 ID 的数据对象；
- `StudyLoadPolicy`：按完成率、睡眠和身体状态计算负荷；
- `StudyProgressLedger`：课程分钟、框架、题目、错题和背诵分别记账；
- 过期 overlay 自动归档，不在对象初始化时修改全局集合。

迁移完成后删除 `@Suppress("UNCHECKED_CAST")` 和初始化副作用。

`StudyPage.kt` 按板块拆为独立文件，但保留一个统一的 `StudyScreenState`，避免 UI 拆完后状态反而更散。

## P6：UI 与性能减脂

优先处理：

1. `StudyPage.kt`：按今日、计划、抽卡、收藏、成就、商店拆分；
2. `VoiceCallPage.kt`：来电态、通话态、音频控制和动画拆开；
3. `RouteActivity.kt`：路由表与 Activity 生命周期分离；
4. `PluginWebViewPage.kt`：WebView 宿主与各插件桥接分离；
5. 聊天消息组件：只对可见消息做富文本和图片重计算。

性能验收指标：

- 首 Token、中位完整回复耗时、工具耗时、Planner、Memory 已有监测；
- 增加聊天页面首帧、滚动掉帧、电话接听到首句、后台主动消息生成耗时；
- Planner、记忆维护和索引刷新不能阻塞可见回复；
- APK 体积、冷启动和内存峰值在每轮瘦身前后对比。

## P7：依赖与发布清理

- Release 已启用 R8 与资源压缩，应保留；
- 分析 APK / AAB，确认 document、web、search、speech 和插件依赖的真实占比；
- 低频大型能力优先做按需初始化，不先盲删依赖；
- 合并可共享的网络配置，但 Ktor 与 Retrofit 只有在确认用途重复后才统一；
- 清理失效的一次性 workflow、历史脚本和过期实验文件；
- CI 失败 issue 使用同一追踪 issue 或自动关闭旧 issue，避免失败记录无限堆积。

## 实施顺序

1. P0：构建、测试、Lint 增量基线；
2. P1：统一回合引擎；
3. P2：拆 `ChatService` 与 `ProactiveMessageService`；
4. P3：中立结构状态 + persona 表达；
5. P4：数字生活真实闭环；
6. P5：学习计划唯一数据源；
7. P6/P7：UI、性能、依赖和发布减脂。

## 每阶段不得回归的指标

- 角色人设一致性；
- 用户纠正优先级；
- 记忆批次与 checkpoint 安全；
- 全局世界书完整注入；
- 普通聊天、电话和主动消息连续性；
- 承诺可执行、可失败、可重试、可取消；
- 学习与生活监督能形成真实后续；
- 角色不声称执行不存在的动作；
- 回复优先展示，后台维护不阻塞主回复。

## 完成定义

项目不是“文件数变少”就算精简。完成应同时满足：

- 核心回合只有一条主链路；
- 巨型服务成为薄协调器；
- 不可达或无真实执行的能力被删除；
- 可选能力按需初始化；
- 构建、测试和 Lint 增量门禁可信；
- 不同人设在相同事实下仍表现出稳定而明显的差异；
- 用户能感受到角色记得、会行动、会履约、会根据反馈调整，而不是只会换一套温柔措辞。
